# Принятые решения и их причины

## 1. Использование slug вместо ID

**Решение:** Все связки работают со slug категорий и атрибутов.

**Причина:** Синхронизация между сайтами мультисайта. ID могут отличаться, а slug остаются одинаковыми.

## 2. Без rewrite rules

**Решение:** Не используем WordPress rewrite_rules.

**Причина:**
- Меньше нагрузка на WordPress
- Проще отладка
- Нет конфликтов с другими плагинами
- Категории уже существуют в WooCommerce

## 3. Три типа виджетов

**Решение:** Universal (основной) + Badges (метки) + Active (активные)

**Причина:**
- Разные сценарии использования
- Разные места размещения (sidebar, header, content)
- Разные настройки атрибутов

## 4. Отдельная настройка для главной страницы

**Решение:** Настройки shop хранятся отдельно от категорий.

**Причина:**
- Главная магазина не является категорией
- Нужны свои атрибуты для фильтрации
- Несортируемый блок в админке

## 5. Drag & Drop в админке

**Решение:** Сортировка через jQuery UI Sortable.

**Причина:**
- Интуитивный интерфейс
- Порядок важен для отображения
- Сортировка категорий и атрибутов внутри них

## 6. Сортировка фильтров: активные → с товарами → пустые

**Решение:** `usort()` с кастомной логикой.

**Причина:**
- Активные фильтры всегда видны
- Доступные фильтры легко найти
- Пустые фильтры не отвлекают

## 7. Неактивные фильтры как `<span>` вместо `<a>`

**Решение:** Пустые неактивные фильтры не кликабельны.

**Причина:**
- Не вводим пользователя в заблуждение
- Визуально показываем недоступность (opacity: 0.5)
- Активные всегда кликабельны (для снятия)
